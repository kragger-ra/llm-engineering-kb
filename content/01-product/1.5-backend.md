# Бэкенд LLM-приложений: async FastAPI, очереди, кэш, наблюдаемость

LLM-вызов — это медленный (секунды), дорогой (токены) и ненадёжный (rate limit,
5xx) сетевой I/O. Бэкенд вокруг него должен: не блокироваться на ожидании
(async), выдерживать всплески и сбои без потери задач (очереди), не платить дважды
за одно и то же (кэш) и быть наблюдаемым (трейсинг каждого вызова). Эта заметка —
про эти четыре слоя и их типичные грабли. Опирается на механику API
(`1.1-provider-apis`) и агентные циклы (`1.3-agents-from-scratch`); мониторинг в
проде (дрейф, стоимость, latency) вынесен в `3.3-production-monitoring`.

## Суть

Веб-фреймворк для LLM-бэкенда почти всегда **асинхронный** (FastAPI/asyncio):
пока один запрос ждёт ответа провайдера, сервер обслуживает сотни других. Долгие и
многошаговые задачи (агенты, батч, генерация документов) выносятся в **очередь** с
воркерами — синхронный HTTP не должен висеть минуту. Очередь даёт надёжность
(at-least-once + идемпотентность), сглаживание всплесков (backpressure) и
изоляцию сбоев (DLQ). **Кэш** на нескольких уровнях срезает стоимость и латентность.
**Наблюдаемость** (OTel + Langfuse) делает каждый вызов трассируемым — иначе отладка
и контроль стоимости невозможны.

## Механика

### async FastAPI: где event loop блокируется

FastAPI на asyncio крутит один **event loop** (цикл событий) в процессе: корутины
по очереди отдают управление на `await` (сетевой I/O), пока ждут. Это даёт высокую
конкурентность на I/O-bound нагрузке (а LLM-вызов — именно она). Правила:

- **`async def` + `await` на каждом I/O** (вызов провайдера через async-клиент,
  БД через async-драйвер) — корректный путь. Управление возвращается в loop на
  время ожидания.
- **`def`-эндпоинт** FastAPI сам выносит в **threadpool** — не блокирует loop, но
  тратит поток. Годится для синхронных библиотек.
- **Смертельный грех: блокирующий/CPU-bound код внутри `async def` без `await`.**
  Он держит единственный loop, и весь сервер встаёт. Симптом — латентность всех
  запросов растёт при одной тяжёлой операции.
- **CPU-bound (токенизация больших текстов, парсинг, эмбеддинг локально)** —
  выносить через `run_in_threadpool` (Starlette) / `loop.run_in_executor`, а тяжёлое
  и долгое — вообще в очередь (отдельный процесс), т.к. GIL не даёт потокам
  параллелить чистый Python.

### Очереди: at-least-once и идемпотентность

Долгие задачи (агентный цикл, батч-обработка) уходят в очередь (Celery — зрелая,
синхронная по корням; **arq** — нативно async, под asyncio-стек). Брокеры дают
**at-least-once delivery**: при сбое воркера/таймауте задача доставляется повторно
→ **дубликаты неизбежны**. Отсюда главное правило: **задача должна быть
идемпотентной** — повторное исполнение не создаёт второй побочный эффект.

- **Idempotency key:** на входящий запрос генерируется/передаётся ключ; результат
  по ключу сохраняется (Redis/БД). Повтор с тем же ключом возвращает сохранённый
  результат, а не делает работу заново. Критично для LLM — иначе платите за токены
  дважды и шлёте дубликаты во внешние системы.
- **arq** реализует «pessimistic execution»: задача не удаляется из очереди, пока
  не завершилась успехом/ошибкой; при падении воркера вернётся в очередь — тем
  важнее идемпотентность.

### DLQ: куда девать «ядовитые» сообщения

**Dead Letter Queue (DLQ, очередь мёртвых писем)** — отдельная очередь для задач,
не обработанных после N ретраев или невалидных по формату («poison messages»). Без
DLQ битая задача ретраится вечно, забивая воркеры. Различай типы ошибок:

- **Транзиентные** (таймаут, rate limit, 5xx) → ретрай с экспоненциальной паузой.
- **Перманентные** (валидация, 4xx, битый вход) → **сразу в DLQ**, не ретраить.

DLQ потом разбирают: чинят причину, реплеят с rate-лимитом и идемпотентностью.

### Backpressure и сглаживание rate limit

Провайдеры лимитируют RPM/TPM; всплеск трафика нельзя слепо переливать в API.

- **Backpressure (противодавление):** ограничивать число задач «в полёте»
  (prefetch/конкурентность воркеров). При переполнении backlog сверх SLO — **отдавать
  429 + Retry-After**, а не копить бесконечно.
- **Экспоненциальный backoff + jitter** на ретраях (1с, 2с, 4с… + случайный сдвиг) —
  гасит «громовое стадо» (thundering herd), когда все клиенты ретраят синхронно.
- **Token-bucket лимитер** перед провайдером — выпускать запросы с темпом ≤ лимита,
  очередь сглаживает пики.

### Кэширование на четырёх уровнях

| Уровень | Ключ совпадения | Когда срабатывает | Инструмент |
|---|---|---|---|
| **Точный (exact)** | хеш точного промпта | повтор идентичного запроса | Redis |
| **Семантический** | близость эмбеддингов запроса | перефразированный тот же вопрос | GPTCache / vector store |
| **Эмбеддингов** | хеш текста → готовый вектор | повторное эмбеддинг одного текста | Redis/БД |
| **Провайдерский prefix** | общий префикс промпта | длинный системный промпт/few-shot | на стороне провайдера/движка |

**Семантический кэш** (GPTCache) эмбеддит запрос и ищет похожий в кэше; совпадение
при близости ≥ порога. По данным GPT Semantic Cache (arXiv:2411.05276): порог
сходства **0.8** даёт оптимум — hit rate **61.6–68.8%** (т.е. до ~69% вызовов API
срезано) при точности положительных попаданий **>97%**. Риск: слишком низкий порог →
«похожий, но не тот» ответ. **Provider prefix caching** (переиспользование KV
системного промпта) разобран в `1.1-provider-apis` и `2.4-inference-serving` —
бэкенд лишь стабилизирует префикс (не менять системный промпт от запроса к запросу).

### Наблюдаемость: OTel + Langfuse, что логировать

Без трейсинга LLM-вызов — чёрный ящик: не видно стоимости, причины плохого ответа,
узкого места латентности. Стандарт — **OpenTelemetry (OTel)** как vendor-neutral
основа трейсов + **Langfuse** как LLM-специфичный слой.

- **Trace = запрос целиком; span = шаг** (вызов LLM, retrieval, tool call). Langfuse
  «generation» — специализированный OTel-span под LLM-вызов с полями `model`,
  параметры, `usage` (токены), `cost`.
- **Логировать:** промпт и ответ (или хеши при чувствительных данных), модель и
  параметры, токены вход/выход и стоимость, латентность по этапам (TTFT отдельно),
  `userId`/`sessionId`/версия промпта, исход (успех/ошибка/ретрай).
- **Пробрасывать атрибуты на все span'ы** (через OTel Baggage), не только на корень —
  иначе не отфильтровать по userId/версии.
- **Сэмплирование** (`sample_rate` 0.0–1.0) на высоком трафике — слать часть трейсов,
  чтобы не платить за хранение всего.

## Практические соображения

- **Граница sync/async/очередь:** быстрый I/O (< ~1с, один вызов) — в async-эндпоинте;
  долгое/многошаговое (агент, батч) — в очередь, эндпоинт возвращает `task_id` +
  поллинг/webhook.
- **Стриминг ответа** — через SSE из async-эндпоинта (`1.1-provider-apis`); не
  аккумулировать весь ответ перед отправкой.
- **Идемпотентность по умолчанию** для всего, что тратит токены или пишет наружу.
- **Версионировать промпты** и логировать версию в трейс — иначе регрессии не
  отследить (`1.4-evaluation`, `3.2-cicd-versioning-tracking`).
- **Кэш инвалидировать** при смене модели/промпта (иначе отдаёте устаревшие ответы).
- **PII:** решить, что логировать в трейсы; маскировать/хешировать чувствительное.

## Режимы отказа

- **Латентность всех запросов растёт от одной операции.** Блокирующий/CPU-bound код
  в `async def` держит event loop. Симптом: p99 скачет при тяжёлом запросе. Фикс:
  `run_in_threadpool`/`run_in_executor`, тяжёлое — в очередь.
- **Дубли действий/двойная оплата токенов.** At-least-once + неидемпотентная задача.
  Симптом: повторные письма/записи при ретраях. Фикс: idempotency key, дедуп
  результата.
- **Воркеры заняты вечным ретраем одной задачи.** Нет DLQ, перманентная ошибка
  ретраится. Симптом: backlog растёт, throughput падает. Фикс: DLQ + различать
  транзиентные/перманентные ошибки.
- **Шквал 429 от провайдера.** Нет backpressure/лимитера, всплеск слит в API.
  Симптом: массовые rate-limit ошибки, каскадные ретраи. Фикс: token-bucket,
  ограничение конкурентности, backoff+jitter, 429+Retry-After клиентам.
- **Семантический кэш отдаёт неверный ответ.** Слишком низкий порог сходства.
  Симптом: «похожий» ответ не на тот вопрос. Фикс: поднять порог (~0.8), эвалить
  hit-качество, сегментировать кэш по контексту.
- **Стейл-ответы после деплоя новой модели/промпта.** Кэш не инвалидирован. Фикс:
  включать версию модели/промпта в ключ кэша.
- **Невозможно отладить плохой ответ в проде.** Нет трейсинга/версии промпта. Фикс:
  OTel+Langfuse, логировать вход/выход/модель/версию/стоимость.

## Код

```python
# FastAPI: быстрый вызов — в async-эндпоинте; CPU-bound — в threadpool; долгое — в очередь.
from fastapi import FastAPI
from starlette.concurrency import run_in_threadpool
app = FastAPI()

@app.post("/chat")                              # I/O-bound: async + await на провайдере
async def chat(req: ChatReq):
    cached = await cache_get(req.idem_key)      # идемпотентность: вернуть готовое
    if cached:
        return cached
    resp = await llm_client.chat(req.messages)  # await -> loop обслуживает другие запросы
    await cache_set(req.idem_key, resp)         # сохранить под ключом
    return resp

@app.post("/parse")
async def parse(req: ParseReq):
    # тяжёлый синхронный парсинг НЕ звать напрямую в async (заблокирует loop):
    result = await run_in_threadpool(heavy_sync_parse, req.text)
    return result

@app.post("/agent")                             # долгое/многошаговое -> в очередь
async def agent(req: AgentReq):
    task_id = await queue.enqueue("run_agent", req.dict(), idem_key=req.idem_key)
    return {"task_id": task_id}                 # клиент поллит статус, не висит на HTTP
```

```python
# Идемпотентная задача очереди (псевдо-arq): повтор безопасен.
async def run_agent(ctx, payload, idem_key):
    if await ctx["redis"].get(f"done:{idem_key}"):   # уже сделано (ретрай/дубль)
        return await load_result(idem_key)
    try:
        result = await do_agent_work(payload)         # реальная работа
    except TransientError:
        raise                                         # ретрай с backoff (транзиентная)
    except PermanentError:
        await send_to_dlq(payload); return None       # в DLQ, НЕ ретраить
    await ctx["redis"].set(f"done:{idem_key}", "1")   # пометить выполненной
    await save_result(idem_key, result)
    return result
```
